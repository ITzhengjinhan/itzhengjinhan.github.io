从一个 url 地址到最终页面渲染完成，发生了什么

总结了六步：

  1. DNS 解析:将域名解析为 IP 地址
  2. TCP 请求连接，即三次握手
  3. 浏览器发送 http 请求
  4. 服务器响应请求
  5. 渲染页面（主要分为 ① 创建 DOM 树、CSSOM 树，② 创建渲染树，③ 计算页面布局，④ 填充颜色，渲染完成）
  6. TCP 断开连接，即四次挥手

get 与 post 请求的区别

  1. get 一般获取数据，post 一般提交数据
  2. get 的参数放在 url 上，post 放在 body 中
  3. 对于 get 请求，浏览器会把 http header 和 data 一并发送过去，服务器响应 200 返回数据
    而对于 post，浏览器先发 header，服务器响应 100，浏览器再发送 data，服务器响应 200 返回数据
    进行说明 post 安全性比 get 高

TCP 和 UDP 的区别

  1. TCP 是面向连接的，UDP 是无连接的
  2. TCP 是可靠性传输，UDP 是不可靠的
  3. 应用场景：TCP：文件传输，UDP:qq 聊天、在线视频

display 的 block、inline 和 inline-block 的区别

  2. inline： 元素不会独占一行，设置 width、height 属性无效。但可以设置水平方向的 margin 和 padding 属性，不能设置垂直方向的 padding 和 margin；

  2. relative（相对定位）：相对原来位置定位，未脱离文档流
  3. absolute（绝对定位）：相对于除 static 定位以外的最近的父元素来定位，脱离文档流，通常使用“子绝父相对”的方式定位，使得子元素相对父元素定位

li 与 li 之间有看不见的空白间隔是什么原因引起的？如何解决？

原因：浏览器默认会把 display 属性为 inline 或者 inline-block 元素间的空白字符（空格换行 tab）渲染成一个空格。也就是我们上面 li 元素换行产生的换行符，而它会变成一个空格，当然空格就占用一个字符的宽度

解决方案：

  1. 为 li 设置 float:left
  2. 将所有 li 写在同一行
  3. 将 ul 内的字符尺寸直接设为 0，即 font-size:0
  4. 给 ul 设置 letter-space:-5px,并且给 li 的 letter-space 设置为 normal

BFC 的含义与运用

BFC：Block Formatting Context，解释为块级格式化上下文，即建立一个隔离的独立容器，容器里面的子元素不会影响外面的元素

一个 HTML 元素形成 BFC，则满足下列的任意条件即可：

  1. float 的值不是 none。（float:left 或者 float:right）
  2. position 的值不是 static 或者 relative。（position:absolute 或者 position:fixed）
  3. display 的值是 inline-block、table-cell、flex、table-caption 或者 inline-flex
  4. overflow 的值不是 visible（overflow:hidden、overflow:scroll）
  5. 父元素与正常文件流的子元素（非浮动子元素）自动形成一个 BFC

BFC 的运用：

  1. 解决 margin 的重叠问题：由于 BFC 是一个独立的区域，内部的元素和外部的元素互不影响，将两个元素变为两个 BFC，就解决了 margin 重叠的问题。
  2. 解决高度塌陷的问题：在对子元素设置浮动后，父元素会发生高度塌陷，也就是父元素的高度变为 0。解决这个问题，只需要把父元素变成一个 BFC。常用的办法是给父元素设置 overflow:hidden。

手写 call、apply、bind

Function.prototype._call = function(context, ...args) {

    // 如果context不传,this就默认绑定到window
    const ctx = context || window;

    // 这里this指调用函数
    ctx.fn = this;
    // console.log(this)

    // 立即执行方法
    const res = ctx.fn(...args);
    delete ctx.fn;  // 为何要delete？ 不删则绑定的ctx对象会添加一个fn方法进来可能产生副作用
    return res;  // 这里为何要return出去结果呢？ _call相当于包装器，ctx.fn函数得到的结果也要return出去
}

Function.prototype._apply = function(context, args) {

    // 如果context不传,this就默认绑定到window
    const ctx = context || window;

    // 这里this指调用函数
    ctx.fn = this;
    // console.log(this)

    // 这里参数为数组
    const res = ctx.fn(...args);
    delete ctx.fn;
    return res;
}

Function.prototype._bind = function(context, ...args) {

    // 如果context不传,this就默认绑定到window
    const ctx = context || window;

    ctx.fn = this;
    // console.log(this)

    return function(...subArgs) {
        const res = ctx.fn(...args.concat(subArgs))
        delete ctx.fn;
        return res;

    }
}

原型与原型链的理解

1.原型：
在 JavaScript 中，原型也是一个对象，原型的作用，则是实现对象的继承。
在 js 的所有函数对象中，都存在一个属性 prototype，该属性对应当前对象的原型。
而所有的 JavaScript 对象，都存在一个proto属性，proto属性指向实例对象的构造函数的原型，理解起来就如下：

var p = new Person();
console.log(p._proto_=== Person.prototype)//true
p是实例对象，Person是p的构造函数，可以看出来p的_proto_属性指向构造函数Person的原型。

2.原型链:在查找一个对象的属性时，会依次从实例本身、构造函数原型、原型的原型上去找，知道找到为止，如果到了原型链顶端，还没找到，则返回 undefined。

3.constructor:constructor 是构造函数创建的实例的属性，该属性的作用是指向创建当前对象的构造函数。
例如，son.constructor == parent;//true

  • 原型继承（prototype）优点：写法简单，容易理解。缺点：无法向父类构造方法传参

 this.name = name;

 this.name = name;
 console.log(name);

 Parent.call(this,name);
}

var  child1= new Child('张三')

  • class 继承（extends,子类的构造函数 必须调用 super）完美解决上述继承存在的不足，缺点：部分浏览器不支持 class 关键字

Cookie、sessionStorage、localStorage、indexedDB 的区别

共同点：都是保存在浏览器端的容器

  1. cookie 只有 4KB 大小，而 sessionStorage 和 localStorage 有 5MB 大小
  2. cookie 常用来保存用户的登录状态，例如将用户 id 存储于一个 cookie 内。其只在设置的 cookie 过期时间之前一直有效
    sessionStorage 仅在当前的浏览器窗口关闭前有效，localStorage 永久有效，除非用户手动清除缓存等。
  3. indexedDB 以键值对存储，可以快速读取，适用 web 场景

// 利用递归函数实现
function copy(obj) {
    // 创建一个空对象用来拷贝之后的内容
    let newObj = {}
    //判断数据是否为 复杂数据类型
    if (typeof obj === 'object') {
        // 是 就将该数据进行 遍历
        for (let k in obj) {
            // 当 obj[k] 的值依然为 复杂数据类型时 重新调用该函数
            // 将obj[k] 当实参 进行调用 直至 obj[k] 的值不在为 复杂数据类型
            newObj[k] = copy(obj[k])
        }
    } else {
        newObj = obj
    }
    // 将 copy 的对象返回出去(不写 return 会得到 undefined)
    return newObj
}

浏览器的缓存你了解过吗

主要分为强缓存和协商缓存（弱缓存）
强缓存：不发起请求，直接使用缓存里的内容，浏览器把 JS,CSS,image 等放在内存中，下次用户访问直接从内存里取，提高性能
协商缓存： 需要向后端发起请求，通过判断来决定是否使用协商缓存，如果请求内容没有变化则返回 304，浏览器就用缓存里的内容

说下事件循环

主线程先执行同步任务，然后才去执行任务队列里的异步任务，先执行微队列后执行宏队列；全部执行完之后等待主线程的调用，调用完之后再去任务队列中查看是否有异步任务，这样一个循环就是事件循环

setTimeout 和 setInterval 最小执行时间

setTimeout 4ms
setInterval 10ms

token 的登陆流程

  1. 客户端登入后，服务器收到请求并校验账户密码
  2. 验证成功后返回 token
  3. 客户端接收 token 后，将其保存在 cookie 中
  4. 客户端每次请求资源/接口时，都要携带这个 token
  5. 服务器收到请求，验证 token 后返回 data 或者 403 等权限失效

经常遇到的浏览器的兼容性有哪些？原因，解决方法是什么，常用 hack 的技巧 ？

  1. 浏览器默认的 margin 和 padding 不同。解决方案是加一个全局的*{margin:0;padding:0;}来统一。
  2. 居中问题,div 里的内容,IE 默认为居中,而在 Firefox 默认左对齐
    解决方法:设置 margin:0 auto;
  3. 块级元素中含有图片时,有点的浏览器会出现图片下有空隙
    解决方法:设置图片的浮动属性 float:left;
  4. 透明性 IE 用 filter:Alpha(Opacity=60)，而其他主流浏览器用 opacity:0.6;

通用 hack 解决方案：条件注释只有在 IE 浏览器下才能执行，这个代码在非 IE 浏览下被当做注释视而不见

例如 IE6 能识别下划线”“和星号” * “，IE7 能识别星号” * “，但不能识别下划线”“，IE6~IE10 都认识”\9”，但 firefox 前述三个都不能认识。

解决过哪些移动端问题

  1. 当设置样式 overflow:scroll 时，ios 上的滑动会卡顿：-webkit-overflow-scrolling:touch;
  2. 移动端文体小于 12px 时异常显示：先把整体放大一倍，然后再用 transform 的 scale 进行缩小;
  3. ios 下 input 按钮设置了 disabled 属性为 true,显示异常：input[type=button]{opcity:1};
  4. ios 下取消 input 输入框再输入英文首字母默认大写：input 标签设置 aotocapitalize=’off’ autocorrect=’off’
  5. 禁用安卓和 ios 选中文字：-webkit-user-select:none;
  6. 禁止 ios 弹出各种窗口：-webkit-touch-callout:none;
  7. 在安卓环境下 placeholder 文字设置行高偏上：input 有 placeholder 属性时候不设置行高；
